# AI Field Ready Enterprise 基本設計書

## 1. 文書概要

### 1.1 目的

本資料は、既存の「SES営業支援SaaS」構想を見直し、**事業会社が社内でAI人材を育成し、実際にAIプロダクト・AI業務改善施策を構築できる状態へ導く教育カリキュラム/育成支援サービス**として再設計するための基本設計書である。

従来のSES向けモデルでは、外部案件への提案やスキルシート補強が主目的だった。一方、本改訂版では、事業会社の内部人材を対象に、AIリテラシー研修ではなく、**自社業務課題を起点に、AIプロダクトを企画・検証・実装・運用できる人材を育成すること**を主目的とする。

### 1.2 対象読者

- 経営者・役員
- DX推進責任者
- 情報システム部門責任者
- 事業部門責任者
- 人事/人材開発部門
- AI/DX推進担当者
- プロダクト開発責任者
- 本サービスの開発・運用担当者

### 1.3 本設計書の位置づけ

本資料は、以下の設計の土台となる。

- サービス概要資料
- UI/UX設計
- 詳細設計
- カリキュラム設計
- API/データ設計
- LP/営業資料
- PoC提案資料

## 2. サービスコンセプト

### 2.1 新コンセプト

**自社の業務課題から、AIプロダクトを作れる社内人材を育てる。**

AI Field Ready Enterpriseは、事業会社向けに、AIリテラシー・プロンプト研修・ツール活用研修に留まらず、自社業務の課題を起点として、AIプロダクト/AI業務改善施策の企画・検証・実装・運用までを推進できる社内AI人材を育成するB2B教育プラットフォームである。

### 2.2 コアメッセージ

> AIを「使える人材」ではなく、AIで「業務とプロダクトを変えられる人材」を育てる。

### 2.3 サービスの提供価値

| 提供価値 | 内容 |
|---|---|
| 業務課題起点 | 汎用教材ではなく、企業ごとの業務課題・データ・既存フローを起点に育成する |
| AIプロダクト構築力 | アイデア出しだけでなく、PoC、要件定義、データ整備、RAG/AIエージェント実装、評価、運用まで学ぶ |
| 実務成果物化 | 学習ログではなく、業務要件定義書、PoC計画、評価レポート、プロトタイプ、運用改善案を成果物として残す |
| 部門横断育成 | DX部門だけでなく、事業部門、情シス、人事、営業企画などが共通言語を持てる |
| 内製化支援 | 外部ベンダー依存を減らし、社内でAI活用テーマを見極め、推進できる状態を作る |

### 2.4 旧SESモデルからの変更点

| 項目 | 旧：SES向けモデル | 新：事業会社向けモデル |
|---|---|---|
| 主対象 | SES企業、若手/待機人材 | 事業会社の社内人材、DX推進担当、事業部門担当 |
| 主目的 | AI/DX案件に提案できる証跡作成 | 自社業務課題をAIで解決できる人材育成 |
| 成果物 | 営業提案サマリー、配属判断資料 | 業務課題定義、AI企画書、PoC、プロトタイプ、評価レポート |
| 利用部門 | SES営業、教育担当 | 経営企画、DX推進、情シス、事業部門、人材開発 |
| 成功指標 | 面談通過、案件化、待機削減 | AI施策創出数、PoC成功率、業務改善効果、内製化率 |
| 主要機能 | 案件要件→育成→証跡→提案 | 業務課題→AIテーマ化→学習→PoC→実装→運用 |

## 3. ターゲット顧客

### 3.1 主要ターゲット

本サービスの主要ターゲットは、AI活用やDX推進の必要性を感じているが、社内にAIプロダクトを構想・実装・運用できる人材が不足している事業会社である。

| セグメント | 企業像 | 課題 | 優先度 |
|---|---|---|---|
| 中堅事業会社 | 従業員300〜3,000名程度 | DX推進部門はあるが、各事業部でAI施策を作れない | 高 |
| 大企業の事業部門 | 複数事業部・部門横断 | AI活用テーマは多いが、推進できる人材が足りない | 高 |
| 製造業 | 生産、品質、保全、設計、営業技術 | 現場データ・文書・図面活用が進まない | 高 |
| 物流/小売/サービス | オペレーション業務が多い | 予測、最適化、業務自動化の内製化が難しい | 中〜高 |
| 金融/保険/不動産 | 文書・審査・顧客対応が多い | 生成AI/RAG活用にガバナンスが必要 | 中 |
| 人材開発部門 | 全社研修企画 | AIリテラシー研修後の実務接続が弱い | 高 |

### 3.2 対象ユーザー

| ユーザー | 役割 | 期待する価値 |
|---|---|---|
| 経営層 | AI活用方針・投資判断 | AI施策が事業成果につながるか判断したい |
| DX推進責任者 | 全社AI/DX推進 | 部門横断でAI人材を育て、PoCを量産したい |
| 事業部門責任者 | 業務課題のオーナー | 自部門の業務改善テーマをAIで進めたい |
| 情シス/IT部門 | 技術基盤・セキュリティ | 安全にAI活用を進め、内製化の技術基盤を作りたい |
| 人材開発部門 | 研修設計・育成管理 | AI研修を実務成果に結びつけたい |
| 受講者 | 社内AI人材候補 | AIを使って自部門の業務改善やプロダクト開発を進めたい |
| メンター/講師 | レビュー・伴走 | 受講者の成果物を評価し、実務品質に引き上げたい |

## 4. 顧客課題

### 4.1 事業会社が直面する主要課題

| 課題 | 内容 |
|---|---|
| AI研修が実務成果につながらない | 生成AIやプロンプトの研修を受けても、自社の業務改善やプロダクト開発に接続できない |
| AIテーマを見極められない | どの業務をAI化すべきか、どこからPoCすべきか判断できない |
| ベンダー依存が強い | AI開発を外部委託しても、社内に判断・評価・運用できる人材が残らない |
| PoC止まりになる | PoCは実施するが、効果検証・運用設計・現場定着まで進まない |
| 事業部門とIT部門の共通言語がない | 業務側は要件を言語化できず、IT側は業務課題を深く理解しづらい |
| ガバナンスが不安 | 生成AI、データ利用、機密情報、著作権、セキュリティの扱いに不安がある |
| 成果が見えない | 研修完了率は見えるが、業務改善効果やAI施策創出数が見えない |

### 4.2 本サービスが解決すること

本サービスは、単なるAI研修ではなく、以下の状態を作る。

- 社内でAI活用テーマを発見できる
- 業務課題をAIプロダクト要件に落とし込める
- RAG、AIエージェント、AI-OCR、データ分析、自動化などの基本構成を理解できる
- PoC計画、評価指標、検証レポートを作成できる
- 外部ベンダーに丸投げせず、要件・評価・運用判断ができる
- 最終的に、部署ごとにAI活用プロジェクトを立ち上げられる

## 5. サービス全体像

### 5.1 サービス名称

仮称：**AI Field Ready Enterprise**

### 5.2 サービス分類

- B2B教育SaaS
- AI/DX人材育成プラットフォーム
- AIプロダクト内製化支援サービス
- 実務課題ベースのAIカリキュラム/伴走支援

### 5.3 サービスの流れ

```text
企業課題ヒアリング
  ↓
AI活用テーマ候補の整理
  ↓
対象部門・受講者の選定
  ↓
ロール別カリキュラム割当
  ↓
基礎学習・実務演習
  ↓
自社課題ベースのPoC設計
  ↓
プロトタイプ作成
  ↓
AI/メンターによるレビュー
  ↓
成果発表・評価レポート化
  ↓
次期AIプロジェクト化/内製化計画
```

### 5.4 中核機能

| 機能 | 概要 |
|---|---|
| 企業課題登録 | 部門ごとの業務課題、利用データ、現行業務フロー、期待効果を登録 |
| AIテーマ診断 | 課題をAI活用テーマに分類し、実現可能性・効果・難易度を可視化 |
| ロール別育成ロードマップ | AIプロダクト企画、RAG、AIエージェント、データ分析、AI-OCR、自動化などのロール別カリキュラム |
| 実務演習 | 自社業務に近いケーススタディ、サンプルデータ、課題演習 |
| PoC設計支援 | 仮説、KPI、評価指標、データ要件、リスクを整理 |
| プロトタイプ提出 | Notebook、API、RAG検証、AIエージェント設計、業務改善案などを提出 |
| AIレビュー | ルーブリックに基づき、提出物の不足点・改善点を自動レビュー |
| メンター承認 | 重要成果物は人間メンターが承認し、実務品質を担保 |
| 成果物レポート | 受講者/部門/プロジェクト単位で成果物をレポート化 |
| 管理者ダッシュボード | 進捗、成果物、AIテーマ数、PoC候補、部門別状況を可視化 |

## 6. 育成対象ロール設計

本サービスでは、事業会社のAI活用を推進するため、以下のロールを育成対象とする。

### 6.1 主要ロール

| ロール | 目的 | 主な対象者 |
|---|---|---|
| AIプロダクト企画人材 | 業務課題をAIプロダクト企画に変換する | 事業企画、DX推進、PdM候補 |
| AI業務改善リーダー | 現場業務を分析し、生成AI/自動化による改善を進める | 事業部門、業務改善担当 |
| RAG/ナレッジ活用推進人材 | 社内文書・FAQ・ナレッジをRAGで活用する | 情シス、DX、総務、人事、CS |
| データ分析/BI人材 | 業務データを分析し、意思決定に使う | 経営企画、営業企画、マーケ、現場管理者 |
| AIエージェント活用人材 | 業務フローをAIエージェントで自動化する | DX推進、情シス、業務企画 |
| AI-OCR/文書自動化人材 | 紙/帳票/文書処理をAIで効率化する | 経理、総務、法務、バックオフィス |
| AIガバナンス推進人材 | 生成AI利用ルール、リスク、評価基準を整備する | 法務、情シス、リスク管理、人事 |

### 6.2 ロール別成果物

| ロール | 成果物例 |

| AI業務改善リーダー | 業務フロー整理、改善提案、プロンプト設計、運用手順書 |
| RAG/ナレッジ活用推進人材 | 文書整理ルール、チャンク設計、検索評価表、回答評価レポート |
| データ分析/BI人材 | 分析Notebook、SQL、可視化ダッシュボード、示唆レポート |
| AIエージェント活用人材 | 業務フロー図、エージェント設計書、API連携案、検証ログ |
| AI-OCR/文書自動化人材 | 帳票項目定義、抽出結果評価、例外処理ルール、精度改善案 |
| AIガバナンス推進人材 | 利用ガイドライン、リスクチェックリスト、評価基準、承認フロー |

## 7. カリキュラム基本設計

### 7.1 カリキュラム設計方針

本サービスのカリキュラムは、以下の考え方で設計する。

1. AIリテラシーで終わらせない
2. 自社課題を教材化する
3. 学習単位を成果物に紐づける
4. 受講者がAIプロダクト開発の全体像を理解する
5. 経営・事業・IT・現場が共通言語を持てるようにする
6. PoC後の運用・ガバナンスまで扱う

### 7.2 カリキュラム全体構造

| フェーズ | 内容 | 代表成果物 |
|---|---|---|
| Phase 0: 初期診断 | 企業課題、部門、受講者スキル、利用可能データを診断 | AI活用テーマ候補一覧、受講者スキル診断 |
| Phase 1: AI/DX基礎 | 生成AI、機械学習、RAG、AIエージェント、データ活用の基礎 | AI活用アイデアシート |
| Phase 2: 業務課題定義 | 現行業務、課題、KPI、制約、関係者を整理 | 業務課題定義書 |
| Phase 3: AIテーマ設計 | 課題をAI施策に変換し、実現可能性・効果を評価 | AIテーマ企画書、PoC計画書 |
| Phase 4: 実務演習 | ロール別にRAG、AI-OCR、データ分析、自動化などを演習 | 演習提出物、レビュー履歴 |
| Phase 5: プロトタイプ開発 | 小規模なAIプロトタイプや業務改善案を作成 | プロトタイプ、検証ログ |
| Phase 6: 評価・改善 | 精度、品質、業務効果、リスクを評価 | 評価レポート、改善計画 |
| Phase 7: 実装/運用計画 | 本番導入に向けた体制、運用、ガバナンスを設計 | 実装ロードマップ、運用設計書 |

### 7.3 標準プログラム例：12週間

| 期間 | テーマ | 内容 | 成果物 |
|---|---|---|---|
| Week 1 | AI活用基礎 | 生成AI、機械学習、RAG、AIエージェントの全体像 | AI活用アイデアシート |
| Week 2 | 業務課題発見 | 業務フロー、課題、ボトルネック、KPI整理 | 業務課題定義書 |
| Week 3 | AIテーマ化 | AIで解くべき課題/解くべきでない課題を判断 | AIテーマ候補リスト |
| Week 4 | データ/ナレッジ整理 | データ、文書、利用制約、品質を確認 | データ/文書棚卸し表 |
| Week 5 | RAG/ナレッジ活用 | 社内文書検索、FAQ、回答評価の設計 | RAG検証計画 |
| Week 6 | データ分析/可視化 | SQL、pandas、BI、業務指標分析 | 分析レポート |
| Week 7 | AI-OCR/文書処理 | 帳票分類、抽出項目定義、精度評価 | OCR検証レポート |
| Week 8 | AIエージェント/自動化 | 業務フロー、API連携、エージェント設計 | 自動化設計書 |
| Week 9 | PoC計画 | 検証範囲、評価指標、成功条件、体制設計 | PoC計画書 |
| Week 10 | プロトタイプ作成 | 簡易デモ、Notebook、RAG評価、業務改善案 | プロトタイプ/検証ログ |
| Week 11 | 評価・リスク整理 | 精度、費用対効果、セキュリティ、運用リスク | 評価レポート |
| Week 12 | 成果発表・実装計画 | 経営/部門向け発表、次期計画 | AIプロジェクト提案書 |

## 13. 非機能要件

### 13.1 セキュリティ

- 企業ごとのデータ分離
- RBACによるアクセス制御
- SSO/SAML対応を将来的に検討
- 監査ログ保存
- 提出物/業務課題の閲覧制御
- 機密情報入力時の注意喚起
- AIレビュー時のデータ送信範囲制御

### 13.2 プライバシー/ガバナンス

- 受講者評価の利用目的を明示
- 人事評価への転用は企業規定に従う
- AIレビュー結果は補助評価として扱い、最終判断は人間が行う
- 生成AI利用ログの保存
- 著作権/機密情報/個人情報に関する教材を含める

### 13.3 可用性/性能

- 教材閲覧、課題提出、レビュー確認は通常業務時間内に安定稼働
- 大容量ファイル提出に対応
- レポート生成は非同期処理
- ダッシュボードは初期表示3秒以内を目標

### 13.4 拡張性

- 企業別カスタム教材
- ロール別カリキュラム追加
- AIモデル/LLMプロバイダー切替
- LMS/HRシステム連携
- Slack/Teams通知連携
- BIツール連携

## 20. まとめ

AI Field Ready Enterpriseは、従来のSES営業支援SaaSではなく、**事業会社が自社内でAI人材を育成し、AIプロダクトやAI業務改善施策を内製化していくための教育・実務伴走プラットフォーム**として再設計する。

本サービスの差別化ポイントは、一般的なAI研修やeラーニングではなく、**自社業務課題を起点に、AIテーマ化、PoC設計、プロトタイプ作成、評価、運用計画までを成果物として残すこと**である。

まずはMVPとして、以下のコア体験を実装する。

1. 企業の業務課題を登録する
2. AI活用テーマに分類する
3. 受講者にロール別カリキュラムを割り当てる
4. 教材学習と実務演習を進める
5. 成果物を提出する
6. AIレビューとメンター承認を行う
7. 成果物レポートとPoC候補を出力する

この流れにより、事業会社は「AIを学んだ人材」ではなく、**自社の業務課題をAIで解決するための社内推進人材**を育成できる。
